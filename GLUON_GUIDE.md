# GLUON_GUIDE

How proto-sqlite and gluon work together, and what the integration
demonstrates about gluon's capabilities.

## What proto-sqlite needs

The repo elevates SQLite into a fully protobuf-encoded interface: the
wire format for `Sqlite.Query` is a typed `SqlStmtList` message whose
structure mirrors the SQLite grammar at the language-spec level. The
only way to get that without years of hand-maintained .proto files is
to *derive* the protos mechanically from SQLite's EBNF grammar.

That's exactly what gluon v2 provides.

## The pipeline

```
lang/sqlite.ebnf
       │
       │  Metaparser.ParseEBNF        (gluon/v2/metaparser)
       ▼
GrammarDescriptor
       │
       │  compiler.GrammarToAST       (gluon/v2/compiler)
       ▼
ASTDescriptor  ──►  CollapseCommaList  (gluon)
                ──►  NameSequence       (gluon)
                ──►  scalarizeX         (proto-sqlite local)
                ──►  Compile + OnMessage (collect MessagePrefix)
                ──►  StripKeywords      (gluon)
                ──►  Compile            (emit FileDescriptorProto)
       │
       ▼
FileDescriptorProto ──► descriptor.ToString ──► sqlite.proto
                     └─► FileDescriptorSet  ──► lang/sqlite.fdset
                     └─► merge/proto.Split  ──► lang/protos/sqlite/*.proto
                     │
                     └─► protoc           ──► sqlite/pb/*.pb.go
                                              sqlite/pb/prefix_map.go
       │
       ▼
Go types used by Sqlite.Query gRPC server (sqlite/server.go)
       │
       │  RenderSQL walks typed proto via protoreflect,
       │  emits MessagePrefix tokens + StringKind values
       ▼
SQL text → exec sqlite3 → QueryResponse
```

All pipeline stages between `ParseEBNF` and `FileDescriptorProto` are
orchestrated by a single ~160-line driver, `lang/cmd/genproto/main.go`.

## Which gluon pieces each stage uses

| Stage | gluon API | File |
|---|---|---|
| Parse EBNF to `GrammarDescriptor` | `metaparser.ParseEBNF` | `lang/cmd/genproto/main.go:46` |
| Lift flat grammar into a tree AST | `compiler.GrammarToAST` | `lang/cmd/genproto/main.go:51` |
| Collapse `x , {"," , x}` into a repeated field with a separator | `compiler.CollapseCommaList` | same |
| Name anonymous sequence wrappers after their leading keywords (`IF EXISTS` → nested `IfExists`) | `compiler.NameSequence` | same |
| Emit `FileDescriptorProto` from the AST | `compiler.Compile` | same |
| Strip leading keyword terminals once their identity is captured by the enclosing message name | `compiler.StripKeywords` | same |
| Report the AST node that produced each emitted message | `compiler.Options.OnMessage` | `lang/cmd/genproto/main.go:66-97` |

Everything gluon provides is reusable across languages — nothing in the
compiler mentions SQLite. The proto-sqlite driver is responsible only
for language-specific concerns (the EBNF source and the scalarizeX
transform).

## What proto-sqlite contributes

Two small pieces of work that gluon alone couldn't do:

### 1. `scalarizeX` — a local AST transform

SQLite's official railroad diagrams render user-supplied identifiers
as oval nodes with no formal production. The EBNF encodes this as
`name = "x"` — a placeholder saying "a string goes here." Gluon's
generic compiler has no way to know that `"x"` in those three
productions (`name`, `string_literal`, `blob_literal`) is different
from `"x"` as a real keyword; it would lower all of them to
`message XKeyword {}` and emit field `XKeyword x_keyword = 1`.

proto-sqlite's answer, at `lang/cmd/genproto/scalarize.go`:

```go
if root.Kind == compiler.KindTerminal && root.Value == "x" {
    return &pb.ASTNode{Kind: compiler.KindScalar, Value: "value"}
}
```

The compiler's new `KindScalar` handling lowers scalar nodes to
`string` fields rather than message references. Net effect:

```proto
message Name         { string value = 1; }
message StringLiteral { string value = 1; }
message BlobLiteral   { string value = 1; }
```

One 30-line transform, composed into a gluon pipeline, replaces what
would otherwise be hundreds of hand-written field overrides.

### 2. `sqlite/render.go` — reflection-based SQL reconstruction

Once the proto tree is typed, we need to serialize an instance back to
SQL text. The renderer walks a `proto.Message` reflectively:

- For each populated field, look up the message's fully-qualified name
  in `sqlitepb.MessagePrefix` (generated by `OnMessage`) and emit the
  recorded leading-keyword tokens.
- Recurse into message-typed fields; emit `StringKind` fields as
  tokens directly.
- Join tokens with conservative SQL spacing.

No per-statement codegen — the renderer is ~100 lines and covers
every grammar production. Adding a new SQL statement is zero work on
the renderer side.

## What this demonstrates about gluon

1. **EBNF → proto is a first-class pipeline, not a toolchain hack.**
   One command (`genproto`) turns a 550-line grammar file into 221
   proto messages plus a prefix map, reproducibly, in under a second.

2. **AST transforms compose cleanly.** `CollapseCommaList`,
   `NameSequence`, and `StripKeywords` each do one thing and return
   new trees. proto-sqlite slots `scalarizeX` into the middle of that
   pipeline without touching any gluon internals — the transforms are
   just functions over `*pb.ASTNode`.

3. **Language-specific conventions live at the edges.** Everything
   gluon ships is grammar-agnostic. proto-sqlite supplies exactly the
   conventions it needs (`"x"` means user string; prefix tokens live
   in a generated Go map) and nothing more.

4. **The `OnMessage` hook enables structural side-channels.** Gluon
   emits messages and proto-sqlite simultaneously harvests leading
   keyword tokens into a Go map — without parsing the emitted
   FileDescriptorProto a second time. The hook is a narrow
   injection point that keeps the two concerns (lowering vs. prefix
   collection) textually independent.

5. **Generated types support reflection-based consumers.** Because
   `compiler.Compile` produces a real `FileDescriptorProto`, the
   generated Go types work with `protoreflect` out of the box. The
   SQL renderer doesn't know about SQLite — it just walks populated
   fields in proto declaration order.

6. **One-way derivation stays stable under grammar edits.** Editing
   `sqlite.ebnf` and re-running `./build.sh` regenerates the protos,
   the prefix map, and any schema-dependent code paths. The typed
   gRPC service keeps working; only callers that reference fields
   that were actually renamed need updating.

## Repo layout, mapped to the pipeline

| Path | Role |
|---|---|
| `lang/sqlite.ebnf` | Source grammar (human-edited) |
| `lang/cmd/genproto/` | Pipeline driver |
| `lang/cmd/genproto/scalarize.go` | proto-sqlite-local AST transform |
| `sqlite.proto`, `lang/protos/sqlite/*.proto` | Generated bundled / split protos |
| `lang/sqlite.fdset` | Serialized `FileDescriptorSet` |
| `sqlite/pb/*.pb.go` | Generated Go types (protoc) |
| `sqlite/pb/prefix_map.go` | Generated leading-keyword map |
| `sqlite/render.go` | Reflection-based SQL serializer |
| `sqlite/server.go` | `Sqlite.Query` gRPC server |
| `sqlite/cmd/server/` | Standalone server main |
| `sqlite/cmd/ripgrpc/` | In-process round-trip demo |

## Running it

```
./build.sh        # regenerate protos + compile
./test.sh         # build + go test ./...
./LET_IT_RIP.sh   # build + test + live query against embedded sqlite
```

All three scripts chain: `LET_IT_RIP.sh` ⊃ `test.sh` ⊃ `build.sh` ⊃
`setup.sh`. There are no other entry points.
